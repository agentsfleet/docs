---

This walks through installing a fleet from the first-party catalogue and seeing it run, starting from a cold machine. We'll use `github-pr-reviewer`, which wakes on a pull request and posts a review comment; the same steps work for any library entry.

<Steps>

<Step title="Install the CLI">
  ```bash
  curl -fsSL https://agentsfleet.dev | bash
  ```

  This installs the `agentsfleet` CLI (it runs npm under the hood, so Node.js is required). Prefer npm directly? `npm install -g @agentsfleet/cli`. Run `curl -fsSL https://agentsfleet.dev` without `| bash` to read the script first.

  Verify:

  ```bash
  agentsfleet --version
  ```
</Step>

<Step title="Sign in">
  ```bash
  agentsfleet login
  ```

  Opens a Clerk-backed approval page in your browser. After you click **Approve**, the page shows a 6-digit verification code — type it back into the CLI to complete login. The code binds the browser approver to the terminal you're typing from.

  Login also fetches your tenant's workspaces and selects the first one as active. Signup auto-provisions a default workspace, so your CLI is ready to install a fleet immediately — no `workspace add` required.

  Hosted execution (event receipts + runs) is **free until July 31, 2026** — no credit card to start; see [pricing](https://agentsfleet.net/pricing) for the metered rates after the trial. You bring your provider and model and pay them directly — agentsfleet marks up zero on inference.

  <Note>
    **No terminal — CI, or an agent driving the CLI?** Skip the browser: supply a token via `--token <token>` or piped on stdin (prefer stdin to keep it out of shell history). For ongoing machine auth, set the `AGENTSFLEET_API_KEY` environment variable instead of logging in — it's sent on every request and overrides a stored login. Add `--no-input` to fail fast instead of waiting for the code. Full precedence in [`agentsfleet login`](/cli/agentsfleet#agentsfleet-login).
  </Note>
</Step>

<Step title="(Optional) Create a named workspace">
  ```bash
  agentsfleet workspace add pr-review
  ```

  Skip this if the signup-provisioned default workspace is fine. Workspaces are the boundary for secrets, access control, and webhook namespaces — every fleet lives in exactly one. Switch later with `agentsfleet workspace use <id>`. See [Command reference → workspaces](/cli/agentsfleet#workspaces).
</Step>

<Step title="Browse the catalogue and add the secret">
  List the first-party library catalogue and the secrets each one needs:

  ```bash
  agentsfleet library
  ```

  ```
  LIBRARY                        NAME                            CREDENTIALS
  github-pr-reviewer             GitHub Pull Request reviewer    github
  security-reviewer              Security reviewer               github
  zoho-sprint-daily-summarizer   Zoho Sprints daily summarizer   zoho
  ```

  `github-pr-reviewer` needs a `github` secret. Add it to the workspace vault before installing (install rejects a bundle whose secrets are missing, [`UZ-BUNDLE-003`](/api-reference/error-codes#fleet-bundle)). Pipe the JSON on stdin so it never lands in shell history:

  ```bash
  agentsfleet secret add github --data=@- <<'JSON'
  { "api_token": "ghp_...", "webhook_secret": "<32-byte hex>" }
  JSON
  ```

  `api_token` lets the fleet post review comments; `webhook_secret` verifies inbound webhook signatures. See [Secrets](/fleets/secrets) for the full vault model.

  Installing from the dashboard instead? The install gallery spells out why each secret is needed — "this fleet needs github to review your pull requests" — and blocks until it's connected, so you can't create a fleet that can't run.
</Step>

<Step title="Install the fleet">
  ```bash
  agentsfleet install --library github-pr-reviewer
  ```

  agentsfleet fetches the entry's pinned source, validates it server-side, prints a requirements preview (credentials, tools, network hosts), then creates the fleet and prints its webhook URL:

  ```
  ✓ github-pr-reviewer is live.
    Fleet ID: 0198a7b2-9e1f-7c3a-8b25-6d4f0a9e2c71
    Webhook URLs (register on the upstream provider):
      github: https://api.agentsfleet.net/v1/webhooks/0198a7b2-9e1f-7c3a-8b25-6d4f0a9e2c71/github
  ```

  Re-running is idempotent. Pass `--name <name>` to back several fleets with one library entry (e.g. one reviewer per repo). See [Fleet library](/fleets/library).
</Step>

<Step title="Register the webhook">
  The CLI prints the webhook URL but does **not** register it for you. Add it to your repository on GitHub — **Settings → Webhooks → Add webhook**:

  - **Payload URL** — the `github:` URL the install step printed.
  - **Content type** — `application/json`.
  - **Secret** — the same `webhook_secret` you stored in the `github` secret.
  - **Which events** — "Let me select individual events" → **Pull requests**.

  Or do it from the terminal with your existing `gh` auth:

  ```bash
  gh api repos/<owner>/<repo>/hooks --method POST \
    -f "name=web" \
    -f "config[url]=https://api.agentsfleet.net/v1/webhooks/<fleet_id>/github" \
    -f "config[content_type]=json" \
    -f "config[secret]=<webhook_secret>" \
    -f "events[]=pull_request" \
    -F "active=true"
  ```

  See [Webhooks](/fleets/webhooks) for signature details and other providers.
</Step>

<Step title="Trigger a run">
  Open a pull request on the repo. Within seconds:

  - GitHub fires `pull_request` to the webhook URL.
  - The fleet wakes, reads the PR, and posts a review comment.

  Tail it — replace `<fleet_id>` with the value the install step printed:

  ```bash
  agentsfleet logs 0198a7b2-9e1f-7c3a-8b25-6d4f0a9e2c71
  ```

  Need to slice by source or time window? `agentsfleet events 0198a7b2-9e1f-7c3a-8b25-6d4f0a9e2c71 --actor 'webhook:*' --since 30m`. Lost track of the ID? `agentsfleet list` prints every fleet in the active workspace. Or watch the same stream in the dashboard at `app.agentsfleet.net`.
</Step>

</Steps>

## What just happened

A fleet is two markdown files: `SKILL.md` (the prose system prompt — what to do, how to phrase a result) and `TRIGGER.md` (the runtime config — when it wakes, which tools it may call, which secrets it reads). The library entry shipped both; the platform stores them server-side. To change behaviour, edit a local bundle and roll it onto the live fleet:

```bash
agentsfleet fleet update 0198a7b2-9e1f-7c3a-8b25-6d4f0a9e2c71 --from ./my-fleet
```

No redeploys, no DAG editor — the next event runs the new prose.
